# Import Export Restore Plan

## Goal

Add a safe import function so a JSON file generated by `/api/v1/export` can restore app data into another Family Menu installation, including Home Assistant add-on installs.

## Scope

- Full overwrite restore only.
- Import source must be the app's own export JSON format with `version: 1`.
- Restore app-owned data:
  - preferences
  - household members
  - dietary profiles
  - meals
  - variation dimensions
  - variation options
  - weekly plans
  - vacation blocks
  - planned meals
  - meal events
  - checklist items
- Do not import or expose deployment runtime options such as Home Assistant database path, seed path, host, or port.
- Merge import is out of scope.

## Backend Tasks

1. Add import request/response schemas.
   - `ImportDataRequest` with exported `data` and `confirm_overwrite`.
   - Reject requests unless `confirm_overwrite` is true.
   - `ImportDataResponse` with counts for imported meals, plans, events, household members, dietary profiles, and checklist items.

2. Add `Store.import_data(data, confirm_overwrite)` with all-or-nothing behavior.
   - Validate top-level export shape and `version`.
   - Build a temporary SQLite database using the current schema.
   - Restore records into the temporary database using existing normalized shapes where practical.
   - Validate foreign key consistency before touching the active database.
   - In a single transaction on the active database, clear app-owned tables and insert restored rows.
   - Roll back on any validation or insertion failure.

3. Add `POST /api/v1/import`.
   - Return `400` for missing confirmation, invalid shape, unsupported version, or failed validation.
   - Return counts and refreshed household summary on success.

4. Keep export compatible.
   - Export format remains `version: 1`.
   - Preserve current field names.
   - Do not add local filesystem paths to export.

## Frontend Tasks

1. Extend API service and TypeScript models.
   - Add export/import interfaces.
   - Add `importData` method.

2. Add Settings UI controls.
   - File picker for `.json`.
   - Show selected file name.
   - Add an import button that opens a confirmation dialog.
   - Confirmation text states that current meals, plans, household config, history, vacation blocks, and checklist state will be replaced.
   - After success, reload Settings data and show imported counts.
   - Show clear errors for invalid JSON, unsupported export, or rejected overwrite.

3. Keep UI compact.
   - Place import/export controls in the existing Planning Defaults action area or a compact portability panel.
   - Avoid displaying database or seed paths.

## Tests

1. Backend API tests.
   - Export then import into a changed database restores meals, household config, plans, vacation blocks, history, and checklist items.
   - Import without confirmation returns `400` and changes nothing.
   - Invalid JSON shape or unsupported version returns `400` and changes nothing.
   - Failed import preserves existing data.

2. Store-level coverage where useful.
   - Validate count summary.
   - Validate foreign-key-linked records survive restore.

3. Frontend verification.
   - `npm run build`.
   - Manual/browser check that Settings can select an export file, warns before restore, and shows success/errors.

## Risks And Compatibility

- Imported data may come from an older export. The first implementation only supports the current `version: 1` shape because that is the only public format.
- Merge import can create duplicates or conflicting IDs, so it remains deferred.
- A full restore in Home Assistant should be done through the UI only for trusted local users; app-level auth is still handled by the deployment layer.
- SQLite transaction handling must avoid leaving the database half-cleared.

## Acceptance Criteria

- A user can export JSON from local development and import it into the HA add-on from the Settings screen.
- Import requires explicit overwrite confirmation.
- A successful import restores the meal catalog, variations, household settings, weekly plans, vacation blocks, history, and checklist state.
- A failed import leaves existing data unchanged.
- Focused backend tests pass.
- Frontend build passes.
